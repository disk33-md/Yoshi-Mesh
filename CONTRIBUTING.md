# Contributing to Yoshi Mesh

Thank you for your interest in contributing to Yoshi Mesh!

## Code of Conduct

We are committed to providing a welcoming and inclusive environment. Please be respectful and constructive in all interactions.

## Getting Started

1. Fork the repository
2. Create a feature branch: `git checkout -b feature/my-feature`
3. Make your changes
4. Write tests for new functionality
5. Ensure all tests pass: `./scripts/test.sh`
6. Submit a pull request

## Development Guidelines

### Code Style
- Follow C++20 best practices
- Use meaningful variable and function names
- Keep functions focused and small
- Add comments for complex logic
- Run `clang-format` before committing

### Testing
- Write unit tests for all new code
- Include integration tests for features
- Maintain >80% code coverage
- Test edge cases and error conditions

### Documentation
- Update relevant documentation
- Add comments to public APIs
- Include examples for complex features
- Keep README.md current

### Commit Messages
Use clear, descriptive commit messages:
```
<type>: <subject>

<body>

<footer>
```

Types: feat, fix, docs, style, refactor, perf, test, chore

## Pull Request Process

1. Update documentation
2. Add tests for changes
3. Ensure CI passes
4. Request review from maintainers
5. Address feedback
6. Squash commits before merge

## Reporting Issues

- Use GitHub Issues for bug reports
- Include steps to reproduce
- Provide system information
- Attach relevant logs

## Feature Requests

- Open an issue with "Enhancement" label
- Describe the use case
- Propose implementation approach
- Discuss with maintainers

## Questions?

Feel free to open a discussion or contact the maintainers.
